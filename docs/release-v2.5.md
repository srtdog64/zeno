# 2.5.0 Release Notes

Status: renderer-facing buffer helper package release.

## Load-Bearing Changes

| Property                         | Status       | Reason                                                                                         |
| -------------------------------- | ------------ | ---------------------------------------------------------------------------------------------- |
| `@exornea/zeno-buffers` package  | load-bearing | Repeated renderer-facing pack helpers now have a narrow dependency-free home.                  |
| Renderer buffer package boundary | load-bearing | The package must not become a WebGL, WebGPU, Three.js, asset-loading, ECS, or scene framework. |
| Renderer case-study evidence     | load-bearing | The fifth package is justified by repeated buffer shapes across public renderer surfaces.      |
| Packed consumer smoke coverage   | load-bearing | The new package must work from tarballs and reject deep imports like the rest of the family.   |
| Renderer-surface benchmark       | diagnostic   | Multi-project metadata scans and pack helpers are benchmark witnesses, not universal proof.    |

## Buffer Package Boundary

`@exornea/zeno-buffers` owns dependency-free fixed-row packing helpers. Its
input is a `DataView`, record count, generated byte length / field offsets, and
caller-owned output typed arrays. Its output is the number of rows packed or
mutated caller-owned buffers.

Routing is explicit: generated scan kernels remain the schema-aware scalar
table-scan surface, while `@exornea/zeno-buffers` owns generic pack and
histogram helpers that write caller-owned typed arrays.

It does not own renderer upload, asset loading, ECS, scene graph behavior,
shader binding, WebGL wrappers, WebGPU binding, or Three.js / Babylon.js
integration. Those layers stay in applications and examples.

## Renderer-Facing Witnesses

The release adds renderer-facing examples for:

- asset catalog rows
- draw batch command rows
- entity transform rows
- grid / dirty-list rows
- sprite atlas rows

The examples deliberately stop at typed arrays that a renderer could consume
later. They do not upload to a GPU or import renderer libraries unless the
example is explicitly a WebGL demo.

## Benchmark Witness

`bench:renderer-surfaces` uses public repository tree metadata from HexGL,
Nemesis, xwing, and NetHack 3D. The fixture stores path hashes, byte sizes,
extensions, and kind tags only. It does not store asset payload bytes.

The benchmark compares JSON metadata ingestion, binary fixed-record scans, and
renderer-facing queue / metadata pack helpers.

## Non-Goals

- No renderer framework package.
- No asset pipeline or loader.
- No ECS or scene graph API.
- No claim of GPU zero-copy upload.
- No schema evolution ABI.
